# OpenClaw Self-Trainer

让 OpenClaw 的 AI 助手从自己的对话中学习，逐步用本地小模型替代昂贵的云端大模型调用。

## 核心理念

OpenClaw 平时正常工作，后台默默收集对话数据、定期训练、自动评估、渐进式地把越来越多的请求卸载给本地小模型。

**增量微调**：不是每次从头训练，而是在上一个 checkpoint 基础上只用新数据继续微调。每天的新数据只被训练一次，避免早期数据被反复训练导致过拟合。

```
Day 1: 基座模型 + 100条新数据 → checkpoint_v1 → 评估
Day 2: checkpoint_v1 + 20条新数据 → checkpoint_v2 → 评估
Day 3: checkpoint_v2 + 30条新数据 → checkpoint_v3 → 评估
...模型能力持续累积，老数据不会反复训练
```

```
用户请求 → [路由器] → 小模型(本地) / 大模型(云端) → 回复用户
                    │
                    └── 持续学习：收集日志 → 训练 → 评估 → 更新路由规则
```

## 架构

### 四个核心模块

1. **数据管道** — 从 OpenClaw session 日志收集、清洗、分类对话数据
2. **SFT 训练** — 用清洗后的数据微调本地小模型
3. **自动评估** — 对比新旧模型，客观评估训练效果
4. **智能路由** — 根据评估结果决定哪些请求可以分发给小模型

### 数据清洗（敏感信息过滤）

训练数据中可能包含真实敏感信息（IP、密码、Token、邮箱等），必须在训练前清除。

**设计原则：**
- 同一真实值 → 同一替换（全局一致，基于 `sha256` 哈希确定性生成）
- 替换值保持上下文合理性（IP 还是 IP 格式、邮箱还是邮箱格式）
- 替换值随机化，避免模型学习固定占位符

**支持的敏感信息类型：**

| 类型 | 说明 | 替换示例 |
|------|------|----------|
| IP 地址 | 公网/内网 IP | `54.123.45.67` → `198.51.170.142` |
| 邮箱 | 个人/服务邮箱 | `user@gmail.com` → `user661@example.org` |
| Token/API Key | 各平台密钥 | `clh_3Ld0bBQ...` → `tk_0625_xxxx...` |
| 密码 | 含特殊字符的密码 | `7Fm842}7dhp{` → `••••••••••••` |
| SSH 公钥 | ed25519/rsa 公钥 | `ssh-ed25519 AAAA...` → `ssh-ed25519 xxxx...` |
| UUID/Session ID | 会话标识符 | `bbdf3f66...` → `aaaa-0e81-4bbb-cccc-...` |
| 服务器路径 | `/home/user` 格式 | `/home/ubuntu/clawd` → `/home/user` |
| 高熵字符串 | 无上下文的长随机串 | `adNHQ9MkV7ZSw...` → `tk_2959_xxxx...` |
| 连接字符串 | DB/Redis 等连接串 | `mongodb://user:pass@host` → `db://user:****@host` |

**使用方法：**

```bash
# 扫描并生成清洗报告
python scripts/clean.py training_data.jsonl

# 输出 training_data_sanitized.jsonl + 控制台报告（含 precision/recall 评估）
```

**实测效果（736条训练数据）：**
- 检出 44 条含敏感信息的记录（33个唯一值）
- Precision: 100%（零误报）
- Recall: ~95%

### 闭环流程

```
┌─────────────────────────────────────────────┐
│                 OpenClaw (运行时)             │
│   用户 → [路由器] → 小模型/大模型 → 回复用户   │
│   所有交互 → 写入日志                         │
└──────────────────┬──────────────────────────┘
                   │ 日志
                   ▼
┌─────────────────────────────────────────────┐
│              数据管道 (定时跑)                │
│   原始日志 → 清洗 → 去重 → 分类 → 训练集     │
└──────────────────┬──────────────────────────┘
                   │ 干净数据
                   ▼
┌─────────────────────────────────────────────┐
│              训练 (有新数据时触发)             │
│   训练集 → SFT 微调 → 新模型 checkpoint     │
└──────────────────┬──────────────────────────┘
                   │ 新模型
                   ▼
┌─────────────────────────────────────────────┐
│              评估 + 决策 (训练后跑)            │
│   新模型 vs 旧模型 vs 大模型                  │
│   → 通过 → 部署                              │
│   → 不通过 → 继续收集数据                     │
│   → 退步了 → 回滚到上一个版本                  │
└─────────────────────────────────────────────┘
```

## 评估体系

不依赖人工评价，基于客观指标：

- **任务成功率** — 是否调用了正确的工具？结果是否正确？
- **幻觉检测** — 是否编造了不存在的信息？
- **工具选择准确率** — 是否选对了工具
- **指令遵循度** — 是否正确理解并执行了用户指令

## 路由器

基于评估结果动态更新路由规则：

```
第一轮训练后：
  工具操作: 85% 准确  → 路由给小模型
  生活闲聊: 91% 准确  → 路由给小模型
  复杂推理: 38% 准确  → 大模型兜底

更多数据训练后：
  复杂推理: 55% 准确  → 继续收集，还不行

最终目标：
  越来越多的请求被小模型处理
  大模型只处理最难的部分
  API 成本线性下降
```

## 时间线

| 阶段 | 时间 | 内容 |
|------|------|------|
| 搭管道 | 第1周 | 清洗脚本、数据集管理、评估脚本 |
| 纯收集 | 第2周 | 只收集数据，不训练 |
| 首次训练 | 第3周 | 第一次 SFT + 评估 |
| 渐进路由 | 第4周 | 部署，只路由最简单的请求 |
| 闭环运行 | 第5-8周 | 逐步扩大路由范围 |
| 稳态运行 | 第2-3月 | 大部分简单请求由小模型处理 |

## 预期效果

- API 成本降低 40-60%
- 简单任务响应更快（本地推理）
- 用户只需偶尔看看周报

```bash
# 2. 让 OpenClaw 读 SETUP.md 并自动配置
# 对 OpenClaw 说："帮我安装 openclaw-self-trainer"
```
